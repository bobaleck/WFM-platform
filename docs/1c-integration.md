# Интеграция с 1С

1С — единственная рабочая внешняя интеграция WFM. Она используется для сверки кадрового статуса сотрудника по ИНН.

## Архитектура

База 1С не публикуется в Web. WFM-сервер работает на Linux, поэтому COMConnector напрямую недоступен.

```text
WFM backend на Linux -> HTTP внутри сети -> Windows 1C Gateway -> COMConnector -> база 1С
```

Windows Gateway — это внутренний агент в локальной сети. Он устанавливается на Windows-сервере с платформой 1С и подключается к базе 1С по обычным параметрам 1С: сервер, имя информационной базы или путь к файловой базе, пользователь и пароль.

## Настройки

Таблица `onec_connection_settings` хранит:

- `connection_mode`: `gateway_http` или `direct_com`;
- `gateway_url`: внутренний адрес Windows Gateway;
- `gateway_token_encrypted`: зашифрованный token Gateway, если используется;
- `infobase_type`: `server` или `file`;
- `onec_server`: сервер 1С для серверной базы;
- `onec_database`: имя информационной базы;
- `onec_cluster`: кластер, если используется;
- `file_base_path`: путь к файловой базе 1С;
- `onec_username`: пользователь 1С;
- `onec_password_encrypted`: зашифрованный пароль 1С;
- timeout, TLS verify, флаги сверки и статус последней проверки.

GET settings не возвращает пароль и token. Возвращаются только флаги `password_saved` и `gateway_token_saved`.

## API WFM

- `GET /api/v1/integrations/onec/settings`;
- `PUT /api/v1/integrations/onec/settings`;
- `POST /api/v1/integrations/onec/check`;
- `POST /api/v1/integrations/onec/diagnose`;
- `POST /api/v1/employees/{employee_id}/check-1c-status`;
- `POST /api/v1/employees/check-all-1c-status`.

`POST /api/v1/integrations/onec/check` выполняет read-only проверку через Gateway: `POST {gateway_url}/api/v1/1c/check-connection`.

`GET /health` Gateway означает только сетевую доступность Gateway. Он не считается успешной сверкой сотрудника и не записывает сотруднику статус `ok`.

## Диагностика

`POST /api/v1/integrations/onec/diagnose` показывает только безопасные признаки:

- ОС сервера;
- режим подключения;
- указан ли Gateway;
- указан ли сервер/база/путь к файловой базе;
- указан ли пользователь;
- сохранён ли пароль;
- доступен ли Direct COM на текущей ОС;
- список недостающих полей.

Пароль 1С, Gateway token и строка подключения с паролем не возвращаются.

## Правило статуса

Сверка выполняется по ИНН физлица:

- есть хотя бы одна карточка сотрудника без даты увольнения — `working`, «Работает»;
- все карточки имеют дату увольнения — `dismissed`, «Уволен»;
- ИНН не найден — `not_found`, «Не найден»;
- ошибка подключения или ответа — `check_error`, «Ошибка сверки».
- Gateway недоступен — `gateway_unavailable`;
- метод Gateway не реализован или ответ некорректен — `gateway_invalid`;
- ошибка подключения Gateway к 1С — `onec_connection_error`.

Статус `ok` запрещён для `employee.onec_status`.

## Поведение без Gateway

Если адрес Gateway не задан, UI показывает: «Укажите внутренний адрес Windows Gateway». Если выбран Direct COM на Linux, backend возвращает: «Direct COM недоступен на Linux. Используйте Windows Gateway.»

## Безопасность

- секреты не хардкодятся;
- пароль 1С и token Gateway хранятся только зашифрованно;
- Gateway должен быть доступен только из внутренней сети;
- firewall Gateway должен разрешать доступ только от WFM-сервера;
- логи не должны содержать пароли, token и строки подключения.
# Актуализация этапа 9.4

1С остаётся источником только кадрового статуса сотрудника по ИНН. WFM не имеет прямого доступа к 1С на Linux и не настраивает Windows/COMConnector.

Сотрудник может быть доступен в нескольких рабочих контурах, но ИНН относится к единой карточке сотрудника WFM. Сверка с 1С не зависит от команды, навыков и Naumen UUID.

Допустимое правило статуса не изменилось:

- если в 1С есть хотя бы одна карточка без даты увольнения — «Работает»;
- если все карточки имеют дату увольнения — «Уволен»;
- если физлицо по ИНН не найдено — «Не найден»;
- если Gateway или 1С недоступны — ошибка сверки.
# Актуализация этапа 9.7

WFM работает на `192.168.12.31`. Сервер 1С и PowerShell Gateway находятся на `10.1.1.12` и обновляются пользователем вручную.

WFM не считает `/health` успешной сверкой сотрудника. Для сотрудника используется только `POST /api/v1/1c/check-employee-status`.

Если Gateway возвращает `found=true`, `status=not_found` и сообщение «Физлицо найдено, но карточки сотрудников не найдены», WFM показывает статус «Карточки сотрудника не найдены». Это не означает увольнение и не означает отсутствие физлица в 1С; это диагностика проблемы поиска карточек сотрудника в Gateway.

Расширенный ответ Gateway сохраняется в `employees.onec_metadata`: версия Gateway, стратегия поиска, карточки, warnings/errors.

Допустимые статусы сотрудника по 1С: `not_checked`, `working`, `dismissed`, `not_found`, `not_found_person_cards`, `check_error`, `gateway_unavailable`, `gateway_invalid`, `gateway_timeout`, `onec_connection_error`.
# Обновление этапа 9.8

Кнопка сверки сотрудника вызывает Gateway endpoint `/api/v1/1c/check-employee-status`, а не `/health`. WFM передаёт в Gateway параметры базы 1С из backend-настроек и ИНН сотрудника. В пользовательской карточке показывается только итоговый статус: "Работает", "Уволен", "Не найден в 1С", "Ошибка сверки", "Не проверялся". Количество карточек и стратегии поиска доступны только в диагностике интеграции.
