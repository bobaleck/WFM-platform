# Архитектура WFM-платформы

Проект является внутренним WFM-слоем для контактного центра «Телесейлз Сервис». На Этапе 2 реализована автономная MVP-версия без внешних интеграций.

## Компоненты

- Frontend: React/Vite cabinet с рабочими разделами WFM.
- Backend: FastAPI API layer, CRUD-логика, auth/RBAC, audit log, демо-данные, сводные отчёты.
- PostgreSQL: основное хранилище сотрудников, команд, навыков, очередей, нагрузки, потребности, смен, графиков, отсутствий, KPI и аудита.
- Redis: инфраструктурный cache/queue placeholder для будущих фоновых задач.
- Scheduler: лёгкий сервис с preview-расчётом потребности по MVP-формуле.
- Nginx: reverse proxy для web UI и backend API.
- Future adapter: место для будущей интеграции с Naumen или другим источником телефонии, без реальных вызовов на Этапе 2.

## Доступ через Nginx

Сайт публикуется через системный Nginx на HTTP port 80:

- `/` -> frontend Vite на `127.0.0.1:5173`;
- `/health` и `/api/` -> backend на `127.0.0.1:8000`;
- `/scheduler/` -> scheduler на `127.0.0.1:8010`.

Backend, frontend и scheduler напрямую открыты только на localhost. Firewall, SSH, mihomo и codex не менялись.

## Интеграция

Раздел «Интеграция» подготовлен как read-only адаптер Naumen REST API `v2/current`.
Настройки хранятся в backend, секреты шифруются и не возвращаются открытым текстом.
API key mode использует `Username` и `X-API-Key`, Basic mode использует HTTP Basic.

Проекты Naumen добавляются вручную по UUID, потому что endpoint списка всех проектов в локальном PDF не подтверждён.
Проверка проекта выполняется через `GET /projects/{uuid}`.
Список сотрудников, отделы, операторы проекта и расписание проекта читаются через подтверждённые read-only endpoints.
Write-запросы в Naumen не выполняются.

## Планировщик MVP

Backend рассчитывает потребность по формуле:

- `workload_seconds = offered_contacts * average_handle_time_sec`;
- `raw_agents = workload_seconds / interval_seconds`;
- `required_agents = ceil(raw_agents / target_occupancy)`.

Черновой генератор графиков берёт пиковую потребность по дате и очереди, исключает сотрудников с отсутствиями, не назначает сотрудника дважды в день и старается не превышать недельный лимит часов.

На Этапе 4 генератор также проверяет обязательные навыки очереди через `queue_skills` и `employee_skills`. Если сотрудников с нужным навыком не хватает, назначение не создаётся, а дефицит отражается в coverage.

## Авторизация и безопасность

Backend использует bearer token и RBAC. Health, version и login остаются публичными, остальные внутренние API требуют авторизацию при `AUTH_ENABLED=true`.

Пароли хранятся как hash. Права назначаются через роли `admin`, `supervisor`, `analyst`, `customer`, `readonly`. Frontend показывает меню согласно правам текущего пользователя.

Security headers добавляются на уровне backend. PostgreSQL и Redis остаются внутри Docker-сети, backend/frontend/scheduler опубликованы только на `127.0.0.1` и доступны пользователю через Nginx.

## Пользовательские роли

- Администратор: полный доступ.
- Супервизор: сотрудники, смены, графики, подтверждение, отчёты и импорт.
- Аналитик: справочники, нагрузка, расчёт, отчёты и экспорт.
- Заказчик: dashboard, опубликованные графики, отчёты и экспорт.
- Только просмотр: dashboard и основные страницы.
# Обновление Этапа 7

Система является кастомным WFM-layer на open-source технологическом стеке. UI адаптирован под внутреннюю систему «Телесейлз Сервис». Планировщик работает как MVP scoring algorithm внутри backend; Timefold не подключён, но может быть рассмотрен позже как отдельный optimization service.

# Обновление Этапа 7.2

Раздел «Интеграция» переведён из заглушки в подготовленный Naumen-модуль без внешних запросов. Добавлены таблицы `naumen_connection_settings`, `naumen_sync_runs`, `naumen_sync_errors`, `external_mappings`; API живёт в `/api/v1/integrations/naumen/*`.

Настройки задаются через UI, секреты шифруются и не возвращаются открытым текстом. Dry run endpoints не сохраняют бизнес-данные и возвращают `not_configured` или `pending`, пока тестовый контур и endpoints не подтверждены.

# Обновление Этапа 8

Добавлены `naumen_projects`, `user_project_access`, `user_preferences` и `naumen_project_schedule_rules`.
Sidebar получил selector активного проекта.
