# Ручной WFM-процесс

С Этапа 9.3 WFM остаётся основной системой ручного ведения сотрудников и справочников. Источники данных: ручной ввод, XLSX/CSV-реестры, сверка кадрового статуса с 1С по ИНН и операционная сверка Naumen по UUID.

## Основной сценарий

1. Администратор создаёт рабочий контур в настройках.
2. Управляющий создаёт команды, навыки, очереди и шаблоны смен.
3. Сотрудники создаются вручную или загружаются через XLSX-реестр.
4. По сотрудникам выполняется сверка с 1С по ИНН.
5. При необходимости сотрудник сопоставляется с оператором Naumen по UUID.
6. Нагрузка загружается через XLSX или CSV по очередям и интервалам.
7. Система рассчитывает потребность операторов по MVP-формуле.
8. Планировщик формирует черновик графика с учётом сотрудников, смен, навыков, отсутствий и потребности.
9. Управляющий проверяет покрытие, подтверждает и публикует график.
10. Отчёты показывают сотрудников, покрытие, план/факт и экспортируют CSV.

## Что ведётся вручную

- рабочие контуры;
- сотрудники;
- команды;
- навыки;
- очереди;
- смены;
- отсутствия;
- настройки планирования;
- импорт нагрузки.

## Что автоматизировано

- проверка ИНН и статуса 1С;
- сопоставление с Naumen по UUID, login или email;
- массовая сверка сотрудников с 1С;
- еженедельная сверка по расписанию после настройки Gateway;
- расчёт потребности из нагрузки;
- черновое формирование графиков;
- пересчёт покрытия.

## Ограничения MVP

- реальный Windows 1C Gateway ещё должен быть развёрнут отдельно;
- Direct COM на Linux недоступен;
- Erlang C пока не реализован, используется MVP-формула;
- графики формируются простым алгоритмом и требуют проверки управляющим.
- endpoint-ы Naumen для статистики сотрудника и интервальной нагрузки требуют уточнения.
# Актуализация этапа 9.4

Ручной WFM-процесс теперь строится вокруг рабочих контуров:

1. Администратор создаёт или выбирает рабочий контур.
2. Сотрудники создаются вручную или импортируются XLSX и получают флаги доступных контуров.
3. Команды создаются внутри конкретного контура.
4. В команду добавляются только сотрудники, которым доступен этот контур.
5. Навыки ведутся глобально для всей платформы и назначаются сотрудникам независимо от проекта.
6. Очереди остаются проектными и используют глобальные требуемые навыки.
7. Нагрузка загружается по очередям, затем считается потребность и формируются графики.
8. 1С сверяет кадровый статус по ИНН.
9. Naumen сопоставляет оператора проекта с сотрудником WFM по ФИО и сохраняет UUID.

СНИЛС и логин Naumen не используются в ручном сценарии.
# Обновление этапа 9.8

Рабочий процесс WFM: создать или выбрать проект, создать сотрудников или загрузить XLSX-реестр, назначить проекты, команду и глобальные навыки, создать очереди и смены, загрузить нагрузку, рассчитать потребность, сформировать черновик графика, проверить покрытие, подтвердить и опубликовать график. 1С используется только для кадрового статуса по ИНН. Naumen используется для операторов проекта и статистики, если endpoint подтверждён.
# Обновление: workflow с Naumen/NCC

Актуальный процесс:

1. Создать или выбрать рабочий контур.
2. В настройках привязать к контуру UUID партнёра Naumen/NCC (`partneruuid`), если нужна автоматическая статистика.
3. Если UUID не указан, контур работает в ручном режиме: нагрузка загружается XLSX/CSV.
4. Обновить очереди и нагрузку из Naumen/NCC или загрузить нагрузку вручную.
5. Рассчитать потребность из локальной WFM-нагрузки.
6. Сформировать график по сотрудникам, навыкам, сменам, отсутствиям и потребности.

Naumen/NCC не является кадровой системой и не увольняет сотрудников. 1С остаётся источником кадрового статуса по ИНН.
