# Отчёт: Naumen/NCC read-only интеграция и предпродовая доработка WFM

Дата: 2026-06-05.

## Что изменено

- Добавлена backend-интеграция Naumen/NCC через read-only PostgreSQL NCC.
- Frontend не подключается к NCC напрямую: все запросы идут только через WFM backend.
- Добавлены env-переменные `NCC_DB_HOST`, `NCC_DB_NAME`, `NCC_DB_USER`, `NCC_DB_PASSWORD`, `NCC_DB_PORT`, `NCC_DB_TIMEOUT_SECONDS`, `NCC_DB_MAX_PERIOD_DAYS` в `.env.example` без секретов.
- Добавлены endpoints `/api/v1/naumen/status`, `/customers`, `/projects`, `/queues`, `/load`, `/employees`, `/operator-workload`, `/forecast-profile`.
- Добавлена синхронизация контура `POST /api/v1/contours/{contour_id}/naumen/sync`.
- Очереди, нагрузка, операторы, операторская нагрузка и профиль прогнозирования сохраняются в локальную WFM БД.
- Массовая сверка Naumen больше не требует ручной предварительной загрузки операторов: backend читает операторов из NCC по `partner_uuid` активного контура.
- Массовая сверка 1С ограничена активным контуром и не отправляет пустые/некорректные ИНН в Gateway.
- Добавлен partial unique index для активных неархивных сотрудников по ИНН.
- “Дашборд” переименован в “Сводка”.
- Sidebar сгруппирован в раскрывающиеся разделы.
- Добавлены loading-состояния для ключевых действий сотрудников, очередей, нагрузки и потребности.
- Добавлена тёмная тема через CSS variables и переключатель в верхней панели.

## Безопасность NCC

- Используются только параметризованные SQL-запросы.
- Произвольный SQL от пользователя не принимается.
- NCC-клиент запрещает write-операции и проверяет обращения только к разрешённым таблицам/представлениям.
- DSN, пароль и полный конфиг подключения не логируются и не возвращаются в API.
- Если NCC env не заполнены, UI показывает: “Интеграция Naumen/NCC не настроена”.
