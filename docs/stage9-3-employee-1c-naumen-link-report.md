# Отчёт Этапа 9.3: сотрудник WFM + 1С + Naumen

Дата: 2026-06-01.

## 1. Модель сотрудника

Сотрудник WFM расширен до связанной модели:

- WFM-карточка — основная карточка сотрудника;
- 1С — кадровый статус по ИНН;
- Naumen — операторский UUID, login, проект и статистика.

Добавлены поля Naumen: `naumen_uuid`, `naumen_login`, `naumen_project_uuid`, `naumen_project_title`, `naumen_status`, `naumen_status_label`, `naumen_last_checked_at`, `naumen_last_check_message`, `naumen_last_sync_at`, `naumen_metadata`.

## 2. Источники данных

- WFM: сотрудники, команды, навыки, смены, графики и отсутствия.
- 1С: только кадровая сверка по ИНН.
- Naumen: операторский UUID, операторы проекта, будущая статистика и фактические выходы.

Naumen не используется как кадровый источник увольнения.

## 3. Исправление ложного ok 1С

`/health` Gateway считается только сетевой доступностью. Сверка сотрудника считается успешной только через `POST /api/v1/1c/check-employee-status` и только если возвращён бизнес-статус. Статус `ok` запрещён для `employee.onec_status`; неожиданные ответы переводятся в ошибку сверки.

Если Gateway отвечает, но метод сверки не реализован, сотрудник получает `gateway_invalid`. Если Gateway недоступен — `gateway_unavailable`.

## 4. Сверка с 1С

Кнопка «Сверить с 1С» вызывает `POST /api/v1/employees/{id}/check-1c-status`, который обращается к Gateway endpoint `check-employee-status`. Результат обновляет поля 1С и не меняет Naumen-идентичность.

## 5. Naumen UUID

В карточке сотрудника добавлены `UUID Naumen` и `Логин Naumen`. UUID используется для поиска локально загруженного оператора Naumen в рамках текущего контура.

## 6. Сверка с Naumen

`POST /api/v1/employees/{id}/check-naumen` проверяет UUID сотрудника по таблице `naumen_operators`. Если оператор найден — статус `linked` или `mismatch`; если UUID не найден — `not_found`; если Naumen не настроен — `api_unavailable`. Fake ok не используется.

## 7. Операторы Naumen

Добавлена таблица `naumen_operators`. Endpoint `POST /api/v1/naumen/operators/sync` загружает операторов проекта через `GET /projects/{projectUuid}/agents`, если endpoint доступен и Naumen настроен.

## 8. Сопоставление

`POST /api/v1/employees/naumen/match` сопоставляет по UUID, login и email. Совпадения по ФИО возвращаются как suggestions и не привязываются автоматически.

## 9. Контуры

Списки сотрудников, команд, навыков, очередей и нагрузки фильтруются по текущему рабочему контуру. Новые сотрудники и справочники получают текущий `project_id`.

## 10. Команды и навыки

В карточке сотрудника команда выбирается из списка, навыки назначаются multi-select. Добавлены backend endpoints для управления составом команды и назначениями навыка сотрудникам.

## 11. Статистика

Добавлены локальные таблицы дневной, интервальной статистики и фактов выхода. API возвращает пустое состояние, если статистика ещё не загружена.

## 12. Нагрузка и потребность

Нагрузка остаётся ручной XLSX/CSV и готовится к источнику Naumen. Импорт нагрузки проверяет очереди в текущем контуре. Потребность считается из нагрузки по MVP-формуле.

## 13. Смены, графики и отсутствия

Смены остаются шаблонами, графики — назначениями сотрудников на смены и очереди, отсутствия блокируют планирование сотрудника. Формы переведены на выбор объектов там, где это реализовано в UI.

## 14. UI

Раздел «Интеграции» снова содержит два независимых блока: «1С — кадровый статус» и «Naumen — операторы и статистика». В сотрудниках добавлены поля Naumen, выбор команды и навыков из списков, а также кнопка «Сверить с Naumen».

- Codex не подключался к реальному серверу 1С.
- Windows Gateway и COMConnector не настраивались.
- Endpoint-ы Naumen для статистики сотрудника и интервальной нагрузки требуют уточнения.
- Планировщик остаётся MVP.
